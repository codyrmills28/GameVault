# Packaged-App Database Migrations — Design

**Date:** 2026-06-27
**Status:** Approved (design), pending implementation plan

## Problem

The packaged Electron app has no database-migration mechanism. `ensureDataDir()`
in `electron/main.js` copies the bundled `template.db` to userData **only on the
first run** (when no live DB exists); for every subsequent launch it reuses the
existing DB as-is. The template is built with `prisma db push` and there are no
migration files in the repo.

When the Prisma schema evolves, existing installs are never updated. This was
observed live: a machine whose DB was created at the v0.1.0 schema (8 tables)
launched a newer build and the dashboard threw

```
PrismaClientKnownRequestError (P2022):
The column `main.Server.healthStatus` does not exist in the current database.
  → prisma.server.findMany() in app/dashboard/page.js   (digest 3766811775)
```

because recent work added `Server.healthStatus` and five new tables
(`MarketplaceTemplate`, `ModInstallation`, `ScheduledTask`, `ServerSnapshot`,
`TemplateVote`) that the live DB lacks.

This is **critical for auto-update**: now that auto-update ships, every release
that changes the schema will break existing installs the same way unless the app
migrates the user's DB on startup.

## Goal

On launch, the packaged app brings the user's database up to the current schema
**before the Next server starts**, preserving existing data. Fresh installs ship
already-migrated. Schema changes are authored as explicit, reviewable migrations.

## Decisions (from brainstorming)

- **Authoring:** adopt `prisma migrate` (explicit per-change SQL migrations);
  stop using `prisma db push`.
- **Application:** a small startup runner that executes pending migration SQL
  through the **already-bundled Prisma query engine** (`$executeRawUnsafe`) — no
  extra engine/CLI bundled.
- **Failure safety:** back up the live DB before applying; apply transactionally;
  on failure leave the original untouched, keep the backup, show a native error
  dialog, and quit without starting the server.

## Non-goals

- Switching off SQLite / supporting Postgres (the schema notes a future Postgres
  path; out of scope here).
- A general rollback/downgrade UI. Migrations are forward-only at runtime.
- Bundling the Prisma migration (schema) engine or the `prisma` CLI into the app.
- Automatic data-loss handling for destructive migrations — destructive SQL is
  authored and reviewed by a developer per change.

## Architecture

### 1. Authoring workflow (dev-time)

Replace the `db push` workflow with `prisma migrate`. Each schema change is
generated with `prisma migrate dev --name <change>`, producing
`prisma/migrations/<timestamp>_<name>/migration.sql` (reviewed in PRs).

Two migrations bootstrap the system:

- **`0001_init`** — the **v0.1.0 schema** (the 8-table state that every existing
  install actually has). Generated by running `prisma migrate dev --name init`
  against `prisma/schema.prisma` as it existed at the `v0.1.0` tag.
- **`0002_<name>`** — the delta from v0.1.0 to today's schema: adds
  `Server.healthStatus` and the five new tables (and any other current diffs).

Rationale for two migrations: existing installs are at the v0.1.0 schema, so the
baseline migration must equal that schema (not the current one) for the
"mark-applied-without-running" baseline (§3) to be correct.

The `prisma migrate dev` step also maintains a local dev DB and `_prisma_migrations`
for the developer; the packaged runtime does not depend on that — only on the
generated `migration.sql` files.

### 2. Startup migration runner — `electron/migrate.js` (new)

A self-contained, dependency-injected module (mirrors the codebase's
`stopAllServers`/`createUpdater` injection pattern) so the decision logic is unit
testable without Electron.

```
runMigrations({ dbPath, migrationsDir, makeClient, copyFile, dialog, now, uuid, log })
  -> { applied: string[], backupPath: string | null }
```

- `makeClient(dbUrl)` returns a Prisma client bound to `dbUrl` exposing
  `$executeRawUnsafe`, `$queryRawUnsafe`, `$disconnect` (the generated client).
- `copyFile(src, dst)` performs the backup copy.
- `dialog` is Electron's `dialog` (for the failure box); injected so tests use a
  mock.

Algorithm:

1. Open a client on `dbPath`.
2. Ensure the tracking table exists:
   ```sql
   CREATE TABLE IF NOT EXISTS "_prisma_migrations" (
     "id" TEXT PRIMARY KEY NOT NULL,
     "checksum" TEXT NOT NULL,
     "finished_at" DATETIME,
     "migration_name" TEXT NOT NULL,
     "logs" TEXT,
     "rolled_back_at" DATETIME,
     "started_at" DATETIME NOT NULL DEFAULT current_timestamp,
     "applied_steps_count" INTEGER NOT NULL DEFAULT 0
   );
   ```
3. Read applied migration names from `_prisma_migrations`.
4. List migration folders under `migrationsDir`, sorted lexicographically (the
   timestamp prefix gives correct order).
5. Compute baseline (§3) — possibly recording `0001_init` as applied without
   running it.
6. `pending` = listed migrations not in the applied set.
7. If `pending` is empty → return `{ applied: [], backupPath: null }` (the common
   case; near-zero overhead).
8. Otherwise: back up (§4), then for each pending migration in order:
   - Split `migration.sql` into statements (see Known constraints).
   - Run them inside a transaction (`BEGIN; … ; COMMIT;` via the client).
   - Insert a record: `id = uuid()`, `checksum = sha256(migration.sql)`,
     `migration_name`, `started_at`/`finished_at = now()`,
     `applied_steps_count = <statement count>`.
9. Return `{ applied: [...names], backupPath }`.

### 3. Baseline for existing installs

A pre-migrations DB has no `_prisma_migrations` table. After creating it (step 2)
it is empty, so we must decide whether the DB already satisfies `0001_init`:

- **Non-empty DB** — detected by the presence of the `User` table
  (`SELECT name FROM sqlite_master WHERE type='table' AND name='User'`): this is a
  v0.1.0-era install. Record `0001_init` as **applied without running its SQL**
  (its tables already exist), then apply `0002+`.
- **Empty DB** (no `User` table) — apply from `0001_init`.

This is safe because v0.1.0 is the only schema shipped before migrations exist
(documented assumption; revisit if a second pre-migration schema is ever
released).

Note: the shipped `template.db` is built by this same runner against an empty DB
(§5), so it already contains a fully populated `_prisma_migrations`. Fresh installs
copy it and the runner is a no-op on next launch.

### 4. Backup & failure handling

- Before applying the first pending migration, copy `dbPath` to
  `userData/backups/realmswap-pre-migration-<YYYYMMDD-HHMMSS>.db`
  (reuse `src/lib/backupPaths.ts` conventions if they fit; otherwise a `backups/`
  dir under userData).
- Each migration runs in a transaction so a mid-migration failure rolls back that
  migration.
- On any thrown error, `runMigrations` does **not** swallow it. `main.js` catches
  it, shows `dialog.showErrorBox("RealmSwap couldn't finish updating",
  "<message>\n\nYour data is safe. A backup was saved to:\n<backupPath>")`, and
  quits — the Next server is never started against a half-migrated DB.

### 5. `main.js` integration

In `app.whenReady()`'s production branch, between `ensureDataDir()` and
`startNextServer()`:

```js
const dataDir = ensureDataDir();
await runMigrations({
  dbPath: path.join(dataDir, "realmswap.db"),
  migrationsDir: path.join(process.resourcesPath, "migrations"),
  makeClient, copyFile: fs.copyFileSync, dialog, now: () => new Date(), uuid: crypto.randomUUID, log: console,
});
serverPort = await getFreePort();
startNextServer(serverPort, dataDir);
```

`runMigrations` rejecting is caught by the existing `whenReady` try/catch, which
already shows an error box and quits — extended to include the backup-path
message. Dev mode (`isDev`) does not run migrations (it uses the dev DB and the
developer's own `prisma migrate`).

### 6. Build changes

- **`scripts/make-template-db.js`**: build `build/template.db` by running the
  **same runner** against a fresh empty DB (apply all migrations from `0001`),
  instead of `prisma db push`. Guarantees the template's schema and
  `_prisma_migrations` match what users converge to.
- **`electron-builder.yml`**: add `prisma/migrations` to `extraResources`
  (`from: prisma/migrations`, `to: migrations`) so the runtime can read the
  `.sql` files at `process.resourcesPath/migrations`.
- **`package.json`**: no `db push` in the build path; document the
  `prisma migrate dev` authoring step. (`postinstall: prisma generate` stays.)

## Components & boundaries

| Unit | Purpose | Depends on |
|------|---------|-----------|
| `electron/migrate.js` | Pure runner: compute pending, baseline, apply SQL, back up, record | injected client/copyFile/dialog/now/uuid |
| `electron/main.js` | Call runner before server start; surface failure dialog + quit | `electron/migrate.js` |
| `scripts/make-template-db.js` | Build template.db via the runner | `electron/migrate.js` |
| `electron-builder.yml` | Bundle `prisma/migrations` | — |
| `prisma/migrations/*` | Authored SQL (`0001_init`, `0002_<name>`, …) | `prisma migrate dev` |

The runner takes all collaborators by injection so its logic (pending-set,
baseline detection, record format, failure path) is unit-testable with mocks and
a temp SQLite DB.

## Error handling

- Migration failure → original DB untouched, backup retained, native error
  dialog, app quits (no server start). Never a half-migrated running app.
- Tracking-table creation is idempotent (`IF NOT EXISTS`).
- A no-op run (no pending migrations) performs no backup and no writes.

## Testing / Verification

- **Unit (vitest)** — `electron/migrate.js` with injected fakes and a temporary
  SQLite DB via the generated client:
  - empty DB → all migrations applied, in order, recorded with checksums.
  - non-empty (v0.1.0-shaped) DB → `0001_init` recorded without running, `0002`
    applied; `User` data preserved.
  - already-migrated DB → no-op, no backup created.
  - a deliberately failing migration → throws, original DB unchanged, backup
    exists.
  - pending-set computation and baseline detection as pure-logic tests with fakes.
- **Manual E2E** (documented): use this machine's stale 8-table userData DB; launch
  the packaged app → it backs up, applies `0002`, the dashboard loads and the
  P2022 error is gone; existing servers/rows are intact.

## Known constraints

- The SQL splitter assumes Prisma-generated DDL: statements terminated by `;` with
  no embedded semicolons. True for generated migrations; not a general SQL parser.
- Baseline assumes the only pre-migration shipped schema is v0.1.0. If another
  pre-migration schema were ever released, baseline detection would need
  revisiting.
- Forward-only at runtime; downgrades are not handled.

## Future / out of scope

- Postgres support.
- A maintenance/repair UI for restoring from a backup.
- Pruning old pre-migration backups.
